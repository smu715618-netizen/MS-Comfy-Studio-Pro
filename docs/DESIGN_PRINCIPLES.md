# 设计原则 (Design Principles)

## 架构原则

### 1. 引擎隔离

```
UI → AI Capability → Scheduler → Engine Adapter → Engine
```

- UI 层不调用任何推理引擎
- 业务层统一通过 Engine Adapter 接口调用
- ComfyUI 仅是默认引擎，可替换
- 未来可添加 Diffusers / ONNX / OpenVINO / CUDA 等

### 2. 模块职责单一

每个模块只负责一个清晰的概念边界：

| 模块 | 职责 | 不包含 |
|------|------|--------|
| `engines/` | 推理引擎适配层 | 不调度、不调用AI能力 |
| `scheduler/` | AI任务调度 | 不直接调用引擎 |
| `ai/` | AI能力接口 | 不处理调度、不操作引擎 |
| `core/resource/` | 资源统一管理 | 不做推理计算 |
| `models/` | 模型生命周期 | 不调用引擎执行 |
| `nodes/` | 节点管理 | 不执行推理逻辑 |
| `plugins/` | 插件系统 | 不含具体业务逻辑 |

### 3. 依赖方向

```
UI ← Capability ← Scheduler ← Adapter ← Engine
       ↑              ↑
     Resource      Config
```

上层依赖下层，禁止反向依赖。引擎模块不引用上层任何模块。

---

## 模块化原则

### 4. 热插拔

所有核心模块必须支持：
- 动态加载（运行时加载）
- 动态卸载（不重启）
- 独立更新（不影响其他模块）

### 5. 按需初始化

禁止：
- 启动时初始化所有引擎
- 启动时扫描所有模型
- 预加载所有插件

必须：
- 首次使用时才初始化
- 使用后保持内存最小占用
- 未使用时立即释放资源

### 6. 配置与代码分离

所有路径、端口、阈值均通过配置文件管理：
- `configs/default.yaml` — 全局默认值
- `configs/xpu.yaml` — 硬件特定配置
- `configs/local.yaml` — 用户覆盖

禁止硬编码路径或参数。

---

## 轻量化原则

### 7. 零开销原则

不用的模块：
- 不初始化 → 不占CPU
- 不加载 → 不占内存
- 不创建进程 → 不占GPU
- 不下载文件 → 不占磁盘

### 8. 渐进增强

基础功能在最低配置下可用：
- 无GPU也能运行（降级到CPU）
- 无ComfyUI也能展示界面
- 无可执行工作流也能保存项目

### 9. 资源回收

使用完的资源必须立即回收：
- 完成推理后释放GPU显存
- 任务完成后关闭临时进程
- 不再需要的索引缓存刷入磁盘

---

## 可扩展原则

### 10. 预留接口

所有未来功能必须先定义接口：
- Engine API → 实现 | 不实现
- AI Capability → 调用 | 返回"未实现"
- Plugin System → 注册 | 不加载

### 11. 多后端兼容

推理层支持多种后端：
- Intel XPU (默认)
- DirectML (兼容)
- NVIDIA CUDA (预留)
- AMD ROCm (预留)
- Apple Metal (预留)
- ONNX Runtime (预留)
- OpenVINO (预留)

### 12. 跨平台构建

核心逻辑不依赖特定OS：
- 路径使用 `pathlib.Path` 而非硬编码
- 平台检测使用 `platform.system()`
- Windows/Mac/Linux使用独立adapter实现

---

## 安全原则

### 13. 隔离执行

- AI推理在独立进程/线程中执行
- 失败不应影响主程序
- 超时自动终止，不阻塞

### 14. 数据隐私

- 不上传用户图片到云端（除非显式确认）
- 模型下载走HTTP而非自定义协议
- 本地日志不记录敏感信息

---

## 开发者原则

### 15. 测试优先

每个公共API必须有对应测试。

### 16. 文档驱动

模块必须有：
- 类/函数文档字符串
- 参数类型注解
- 返回值说明

### 17. Git 提交规范

```
Phase x.y: 模块名称

详细描述（如本报告所示）
```

---

> **记住**: 我们的目标是打造一款真正的专业修图软件。  
> 底层技术只是手段，用户体验才是目的。
